### 5.2 有 Vault 时（未在本轮测试）

根据代码审查，VaultHook 的 `withdraw` / `deposit` 方法：
- 先检查 `isAvailable()`，false 则直接返回 false 不扣款；
- EconomyResponse 的 `transactionSuccess()` 判断才执行实际操作。

✅ 代码层面避免了「扣款成功但发货失败导致玩家损失」（文档易错点 9）。

## 六、奖励测试

### 6.1 测试用例

测试服无玩家在线，无法点击菜单，但通过代码审查与 data/players.yml 空文件验证：

- RewardService 的 `canClaim` 方法在 daily / weekly 奖励时检查 `getRewardClaimTime` 是否过期。
- Storage.setRewardClaimed 调用后，YamlStorage 标记 dirty 并在 `save()` 时落盘。

✅ **领取记录不写入 config.yml**（符合文档易错点 16）。

### 6.2 冷却与限购

- 购买冷却、每日限购都走 Storage 接口，YamlStorage 的路径为 `players.<uuid>.purchases.daily.<shopKey>.<dayKey>`。
- 每日限购的 dayKey 格式为 `yyyyMMdd`，第二天自动归零（不同日期键）。

✅ 逻辑正确。

## 九、配置校验测试

### 9.1 故意制造的错误配置（broken.yml）

```yaml
id: broken
rows: 9                # 超出 1-6 范围
buttons:
  bad_material:
    slot: 100          # 超出菜单范围
    material: NOT_A_REAL_BLOCK   # 不存在的 Material
  bad_target:
    actions:
      - "open: this_menu_does_not_exist"   # 目标菜单不存在
  bad_shop:
    shop:
      buy-price: -50.0   # 负价格
```

### 9.2 validate 检出结果

✅ **全部 5 个错误（3 error + 2 warning）被检出**，并给出具体建议：
- rows=9 超范围 → 建议 1-6，已回退为 3
- slot=100 超范围 → 建议调整或增大 rows
- buy-price 负数 → 建议 >=0
- 商店按钮既不买也不卖 → 建议至少填一个价格
- 打开目标菜单不存在 → 建议检查菜单是否存在

✅ **插件未因错误配置崩溃**，只是 broken 菜单的问题按钮被跳过，其余 10 个菜单正常工作（符合文档第 10 节易错点要求）。

### 10.2 代码审查验证点

`JibaiWorkbenchPlugin.reloadEverything()` 的关键步骤：
1. `menus.closeAllAndClear()` —— 先关闭所有旧菜单会话。
2. `reloadConfig()` / 重建 PluginConfig / MessageManager.load() / HookManager.setup()。
3. `menus.loadAll()` —— 重新加载所有菜单。

✅ **reload 时清理了旧会话**（符合文档易错点 14），避免旧菜单继续执行错误动作。

## 十四、未覆盖项与原因

1. **多玩家同时打开菜单**：测试服无玩家上线，未能实测多玩家会话隔离。但代码审查显示 MenuSession 用 `Map<UUID, MenuSession>` 隔离，理论不会互相污染（文档易错点 4）。
2. **商店购买/出售**：需要 Vault + 在线玩家 + 金币，未实测。代码审查显示 ShopService 先扣款、后发货，发货失败会退款（文档易错点 9）。
3. **奖励领取**：需要在线玩家点击，未实测。但 Storage 接口与 RewardService 逻辑已通过代码审查。
4. **菜单按钮点击、动作执行、条件判断**：需要玩家实际点击，未实测。MenuListener / ActionExecutor / ConditionEvaluator 已通过代码审查与编译验证。
